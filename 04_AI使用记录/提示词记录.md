## 四、项目业务流程总结

### 1. 用户登录流程

```text
用户打开登录页
→ 获取验证码
→ 输入账号、密码和验证码
→ 后端校验登录信息
→ 返回 JWT 和用户信息
→ 前端保存 Token
→ 后续请求自动携带 Authorization 头
```

### 2. 学生提交工单流程

```text
学生登录
→ 进入新建工单页面
→ 填写标题、位置、分类、优先级、描述等信息
→ 前端提交工单并携带 Idempotency-Key
→ 后端创建工单
→ 学生可在我的工单中查看进度
```

### 3. 管理员处理工单流程

```text
管理员登录
→ 进入工单管理
→ 查询和筛选工单
→ 审批通过或驳回
→ 为工单分配维修人员
→ 查看统计数据
```

### 4. 维修人员处理任务流程

```text
维修人员登录
→ 查看我的任务
→ 接单或拒单
→ 处理完成后提交完成信息
→ 工单状态更新
```

### 5. 管理员维护用户流程

```text
管理员进入用户管理
→ 按用户名、真实姓名、手机号、角色、状态、时间筛选
→ 新增用户或修改用户资料
→ 重置密码、启用禁用、删除用户
→ 批量处理用户
```

本节用于补充本项目从功能开发、界面优化、部署验证到文档整理的完整 AI 协作过程。以下内容根据实际多轮对话整理，删除了敏感配置、账号密钥、Token 等信息。

### 1. 工单图片上传、删除与预览

用户最初提出希望“学生端新建工单可以上传图片，管理员端详情可以查看图片，我的工单详情也可以查看图片”。AI 根据该需求检查前后端文件上传逻辑，并围绕 OSS 上传接口完成了工单图片能力的补充。

主要处理内容：

- 学生端新建工单支持上传现场图片。
- 管理员端工单详情支持查看现场图片。
- 学生端“我的工单”详情支持查看图片。
- 修复上传图片后再删除，仍然把已删除图片地址保存到数据库的问题。

## 十一、项目经验与后续建议

根据两个补充文档与当前项目实际开发过程，整理出以下经验：

- 前后端接口路径必须完全一致，例如 `/api/work-orders/page` 和 `/api/work-order/page` 只差一个 `s`，也会导致 404 或 `No static resource`。
- Spring Boot Controller 的分页参数建议写成 `@RequestParam("pageNum")`，不要只写 `@RequestParam int pageNum`。
- MyBatis-Plus 分页需要 Mapper 继承 `BaseMapper<T>`，并配置分页插件。
- OSS 上传成功不等于图片一定能访问，Bucket 权限需要允许公网读，或者由后端生成临时签名 URL。
- 头像 URL 应保存到数据库，不应只保存到 localStorage，否则换设备或重新登录后会丢失。
- 登录接口返回什么，前端 Pinia 才能存什么，所以 `LoginVO` 需要包含 `avatarUrl`、`phone` 等字段。
- 管理端分配维修人员应使用下拉选择，显示 `ID - 姓名`，避免手动输入 ID 出错。
- 前端全量过滤只适合小数据，正式系统应使用后端分页和后端条件查询。
- Redis 是否配置到 Windows 全局环境变量，不影响 Spring Boot 通过 `host:port` 连接 Redis。
- 文档归档时应保留提示词、处理过程和结果说明，但不应保留真实密钥、Token、数据库密码等敏感信息。

后续可继续完善的方向：

- 工单管理和用户管理分页进一步统一为后端分页。
- 工单详情加入更多附件预览能力，例如图片压缩、裁剪和批量上传。
- 维修人员派单可以加入按工种、区域、空闲程度的智能推荐。
- 首页仪表盘可以继续加入趋势图、状态分布图和维修效率统计。
- 用户中心可以继续完善手机号修改、邮箱绑定和密码强度校验。
- WebSocket 或消息队列可用于实现工单状态实时通知。
- k6 压测 Skill 可继续扩展为自动启动服务、自动获取 Token、自动生成报告。
